# MyHarness 발표용 설명 자료

작성일: 2026-06-11

## 1. 한 줄 소개

MyHarness는 사내 구성원이 AI Agent를 더 쉽게 체험하고, 업무에 적용하고, 그 과정을 투명하게 확인할 수 있도록 만든 브라우저 기반 AI Agent 업무 지원 시스템입니다.

## 2. 왜 필요한가

최근 AI는 단순히 질문에 답하는 챗봇을 넘어, 파일을 읽고, 자료를 찾고, 코드를 고치고, 보고서를 만들고, 여러 단계의 업무를 스스로 수행하는 Agent 형태로 발전하고 있습니다.

하지만 실제 조직 안에서 AI Agent를 도입하려면 몇 가지 장벽이 있습니다.

- 사용자가 명령어 기반 도구에 익숙하지 않으면 접근하기 어렵습니다.
- Agent가 내부에서 무엇을 하고 있는지 보이지 않으면 신뢰하기 어렵습니다.
- 결과물이 어디에 저장됐는지, 어떤 파일을 읽었는지, 어떤 작업 단계를 거쳤는지 확인하기 어렵습니다.
- 사내 업무에 필요한 스킬, 데이터 연결, MCP, 보고서 양식 등을 매번 새로 설명해야 합니다.
- 부서나 프로젝트별 작업 기록과 산출물을 체계적으로 관리하기 어렵습니다.

MyHarness는 이런 문제를 해결하기 위해 만들어졌습니다. 핵심 목표는 “AI Agent를 전문가만 쓰는 도구가 아니라, 사내 구성원이 업무 도구처럼 사용할 수 있게 만드는 것”입니다.

더 본질적으로는, 우리 회사가 AX를 위해 직접 축적해야 할 두 가지 자산을 만들기 위한 플랫폼이기도 합니다. 하나는 사내 데이터와 시스템을 연결하는 MCP이고, 다른 하나는 현업 도메인 지식을 AI가 실행 가능한 형태로 정리한 Skill입니다. MCP는 DX 관련 전문부서가 표준 인터페이스와 보안 체계에 맞춰 만들 수 있습니다. 하지만 Skill은 현업 직원들이 참여해야 합니다. 실제 업무의 판단 기준, 예외 처리, 보고 방식, 검토 포인트는 업무를 하는 사람들 안에 있기 때문입니다.

따라서 MyHarness의 중요한 목적 중 하나는 직원들이 Skill을 만들고, 실제 업무에 써보고, 점검하고, 테스트하고, 수정하는 cycle을 돌릴 수 있게 하는 것입니다.

```text
Skill 제작 cycle = 만들기 → 사용해보기 → 점검하기 → 테스트하기 → 수정하기
```

AI Agent 자체는 MyHarness가 아니어도 언젠가 누군가는 만들 수 있습니다. 하지만 우리 회사가 AX를 제대로 하려면 우리 업무에 맞는 Skill과 MCP를 계속 축적해야 합니다. MyHarness는 그 축적이 실제로 일어나는 사용·제작 플랫폼을 목표로 합니다.

## 3. MyHarness가 하는 일

MyHarness는 사용자가 브라우저에서 AI Agent와 대화하면서 실제 업무를 수행할 수 있게 해줍니다.

대표적으로 다음과 같은 일을 할 수 있습니다.

- 문서, 데이터, 코드, 웹 자료를 읽고 분석합니다.
- 보고서, 요약문, 대시보드, HTML 산출물 등을 생성합니다.
- 현재 작업 폴더의 파일을 참조해 맥락에 맞는 답변을 만듭니다.
- Agent가 실행한 작업 단계와 도구 사용 내역을 화면에서 보여줍니다.
- 생성된 파일을 오른쪽 패널에서 바로 미리볼 수 있습니다.
- 프로젝트별 작업 공간과 대화 기록을 유지합니다.
- 사내 업무에 맞춘 스킬, MCP, 플러그인을 연결할 수 있습니다.

즉, MyHarness는 단순 채팅창이 아니라 “AI가 실제 업무 파일과 도구를 다루는 작업 공간”입니다.

## 4. 핵심 관점: AI가 업무를 하려면 무엇이 필요한가

MyHarness를 설명할 때 가장 중요한 관점은 “충분히 똑똑한 AI 모델만 있으면 모든 것이 자동으로 해결된다”가 아니라는 점입니다.

AI가 실제 업무를 수행하려면 먼저 세 가지 핵심 재료가 필요합니다. 충분히 똑똑한 지능인 LLM, 회사의 업무 지식인 Skill, 사내 정보에 접근하는 MCP입니다. 그리고 이 세 가지가 따로 흩어져 있으면 사용자는 여전히 직접 복사하고, 붙여넣고, 저장하고, 검토해야 합니다.

Skill이 필요한 이유는 사람에 비유하면 쉽습니다. 아무리 머리가 좋은 사람이 있어도 우리 회사에 처음 와서 바로 업무를 하라고 하면 어렵습니다. 내부 용어, 업무 절차, 보고 방식, 판단 기준, 예외 처리, 검토 포인트를 모르기 때문입니다. LLM도 마찬가지입니다. 지능은 뛰어나도 우리 회사의 업무 도메인 지식이 없으면 실제 업무를 바로 수행하기 어렵습니다. Skill은 이 업무 지식을 AI가 읽고 실행할 수 있는 형태로 정리한 것입니다.

여기서 Harness의 역할이 중요합니다. Harness는 LLM, Skill, MCP와 나란히 놓이는 네 번째 부품이라기보다, 세 요소를 감싸서 실제 업무 환경으로 만들어주는 실행 레이어입니다.

| 요소 | 의미 | 역할 |
|---|---|---|
| 충분히 똑똑한 지능 | LLM | 판단, 추론, 문서 작성, 계획 수립 |
| 업무 도메인 지식 | Skill | 우리 회사의 보고서 형식, 검토 기준, 업무 절차, 산출물 품질 기준 |
| 사내 정보 접근 인터페이스 | MCP | 내부 데이터, 시스템, 문서, 지식베이스, 외부 전문 데이터와 연결 |
| 실행과 관리를 감싸는 환경 | Harness | LLM·Skill·MCP를 하나의 업무 화면, 작업 기록, 권한, 파일 관리, 산출물 미리보기, 도구 사용, closed-loop 업무 흐름으로 묶음 |

이 구조가 갖춰지면 AI는 단순히 “답변을 생성하는 도구”를 넘어, 실제 업무를 수행하는 Agent가 될 수 있습니다.

여기서 Harness의 중요한 역할 중 하나가 Agentic Workflow를 가능하게 하는 것입니다. AI가 한 번 답변하고 멈추는 것이 아니라, 스스로의 행동 결과를 보고 다음 행동을 정할 수 있어야 합니다.

```text
Observe → Plan → Tool-use → Result → Reflect/Revise
```

이 closed loop 안에서 Tool-use가 핵심이 됩니다. 검색, 파일 읽기·쓰기, 코드 실행, preview 확인, MCP 조회 같은 도구를 AI가 상황에 맞게 사용하고, 그 실행 결과를 다시 맥락으로 받아들일 수 있어야 실제 업무 수행으로 이어집니다. 즉, Harness는 LLM·Skill·MCP를 담는 껍데기만이 아니라, AI가 보고, 행동하고, 결과를 확인하고, 다시 고치는 순환 구조를 운영하는 환경입니다.

다시 말해 MyHarness의 핵심 의의는 다음 구조로 정리할 수 있습니다.

```text
Harness
└─ 충분히 똑똑한 LLM
└─ 업무 도메인 지식이 담긴 Skill
└─ 사내 정보와 시스템에 접근하는 MCP

= AI가 실제 업무를 수행할 수 있는 환경
```

이론적으로 이 구조가 충분히 성숙하면, AI는 사람이 하던 상당수의 지식 업무를 대신하거나 함께 수행할 수 있습니다. 보고서 작성, 자료 조사, 데이터 분석, 문서 비교, 요약, 기획안 작성, 코드 수정, 업무 시스템 조회 같은 작업이 모두 이 틀 안으로 들어올 수 있습니다.

따라서 AX, 즉 AI Transformation을 제대로 추진하려면 단순히 최신 모델을 도입하는 것만으로는 부족합니다. 회사가 집중해야 할 것은 “모델을 업무에 붙이는 연결 구조”입니다. 업무 지식은 Skill로 만들고, 사내 정보 접근은 MCP로 열고, 이 둘과 LLM을 Harness 안에서 안전하고 편리하게 운영해야 합니다.

미래에 지금보다 훨씬 뛰어난 LLM이 나오더라도 Skill과 MCP는 계속 필요합니다. 모델의 일반 지능이 좋아지는 것과 우리 회사 업무를 아는 것은 다른 문제입니다. 또한 사내 데이터와 시스템에 안전하게 접근하는 문제도 모델 성능만으로 해결되지 않습니다. 따라서 우리 회사의 AX 핵심은 특정 모델 하나가 아니라, 우리 업무에 맞는 Skill을 축적하고 MCP를 연결하는 역량입니다.

## 5. 일반 챗봇과의 차이

일반 챗봇은 주로 사용자의 질문에 답변을 생성합니다. 반면 MyHarness는 답변 생성뿐 아니라 실제 작업 흐름을 지원합니다.

| 구분 | 일반 챗봇 | MyHarness |
|---|---|---|
| 사용 방식 | 질문과 답변 중심 | 업무 수행 중심 |
| 작동 구조 | 한 번의 답변 생성 중심 | Observe→Tool-use→Result closed loop |
| 파일 처리 | 제한적 또는 수동 첨부 중심 | 작업 폴더의 파일을 직접 참조 |
| 결과물 | 대화창 안의 텍스트 중심 | HTML, Markdown, PDF, 이미지 등 파일 산출물 |
| 작업 과정 | 내부 과정이 잘 보이지 않음 | 단계별 작업, 도구 사용, 진행 상태 표시 |
| 확장성 | 모델 기능에 의존 | 스킬, MCP, 플러그인으로 업무별 확장 |
| 기록 관리 | 대화 단위 기록 | 워크스페이스별 기록과 산출물 관리 |

이 차이 때문에 MyHarness는 단순 질의응답보다 “반복 업무 자동화”, “보고서 작성”, “자료 조사”, “프로젝트 보조”, “사내 AI Agent 교육”에 더 적합합니다.

## 6. P-GPT와 MyHarness는 무엇이 다른가

우리 회사에는 이미 P-GPT가 있기 때문에, 많은 구성원은 “이미 회사용 AI가 있는데 MyHarness가 왜 필요한가?”라고 생각할 수 있습니다.

P-GPT는 사내에서 AI를 사용할 수 있게 해주는 중요한 기반입니다. 특히 보안과 인증이 갖춰진 환경에서 LLM을 사용할 수 있다는 점은 큰 장점입니다. 다만 P-GPT의 중심은 여전히 “사용자가 질문하고 AI가 답변을 출력하는 채팅형 AI”에 가깝습니다.

반면 MyHarness는 P-GPT 같은 채팅형 AI를 업무 관점에서 한 단계 개선한 도구입니다. 답변 생성뿐 아니라 업무 실행, 산출물 관리, 스킬, MCP까지 함께 묶는 업무 실행형 AI 환경으로 볼 수 있습니다.

| 구분 | P-GPT | MyHarness |
|---|---|---|
| 기본 성격 | 회사용 AI 채팅 서비스 | AI Agent 업무 수행 환경 |
| 사용 방식 | 사용자가 질문하고 답변을 받음 | 사용자가 목표를 주면 Agent가 단계적으로 작업 수행 |
| 업무 지식 적용 | “나의 GPT” 커스텀 프롬프트 중심 | 상황에 맞는 Skill 선택·추천·호출, routing, 스크립트/파일 연계 가능 |
| 인터넷 검색 | 사용자가 별도 검색 기능을 클릭해야 함 | LLM 지식이 부족하다고 판단하면 Agent가 도구로 검색 가능 |
| 사내 정보 접근 | ECM 연동이나 첨부 기반 활용 가능 | MCP, VectorDB, 전문 API 등 구조화된 연결로 확장 가능 |
| Agentic Workflow | 질문 후 답변 중심 | 도구 사용 결과를 다시 보고 다음 작업을 이어가는 closed loop |
| 산출물 처리 | 대화창에 결과 출력 | 파일 생성, 저장, 미리보기, 수정, 재생성, 공유 흐름 지원 |
| HTML/대시보드 | 원문 코드 출력 중심, 렌더링 확인 어려움 | HTML을 실제 화면으로 preview하며 수정 가능 |
| 작업 과정 | 내부 수행 과정이 제한적으로 보임 | 도구 사용, 작업 단계, 생성 파일을 투명하게 표시 |
| 확장 방향 | 좋은 프롬프트를 잘 쓰는 방식 | 업무 지식과 시스템 연결을 플랫폼화하는 방식 |

### 6.1 “나의 GPT”와 MyHarness Skill의 차이

P-GPT의 “나의 GPT”는 커스텀 프롬프트를 저장해 특정 방식으로 답변하도록 유도할 수 있습니다. 이 점에서 겉으로는 MyHarness의 Skill과 비슷해 보일 수 있습니다.

하지만 MyHarness의 Skill은 단순한 프롬프트 저장소보다 넓은 개념입니다.

- 상황에 맞는 Skill을 Agent가 선택·추천·호출하도록 설계할 수 있습니다.
- Skill 설명, 사용 조건, 작업 절차, 산출물 기준을 구조화할 수 있습니다.
- 단순 문구 지시가 아니라 `.py` 스크립트, reference 파일, template, asset과 연계할 수 있습니다.
- routing을 통해 특정 업무는 특정 Skill이나 MCP 흐름으로 보내도록 설계할 수 있습니다.
- 조직의 반복 업무 노하우를 파일 단위로 관리하고 버전 관리할 수 있습니다.
- Skill이 쌓일수록 개인 프롬프트가 아니라 회사의 업무 자산이 됩니다.

즉, “나의 GPT”가 개인화된 답변 스타일과 지침에 가깝다면, MyHarness Skill은 업무 절차와 도구 사용법을 AI가 실행 가능한 형태로 정리한 운영 자산에 가깝습니다.

이 때문에 Skill은 IT나 DX 전문부서만의 산출물이 아닙니다. MCP가 시스템 연결을 위한 전문 인프라라면, Skill은 현업이 자기 업무를 문서화하고 검증하며 발전시키는 영역입니다. 좋은 Skill은 한 번에 완성되기보다, 실제 사용 과정에서 “만들고, 점검하고, 테스트하고, 수정하는” 반복 cycle을 통해 성숙합니다. MyHarness는 이 cycle을 돌릴 수 있는 플랫폼이어야 합니다.

### 6.2 검색과 정보 접근 방식의 차이

P-GPT에서도 인터넷 검색이나 ECM 연동을 사용할 수 있지만, 사용자가 별도로 기능을 선택하거나 자료를 첨부하는 방식에 가깝습니다.

MyHarness는 Agent가 업무를 수행하다가 모델의 기본 지식만으로 부족하다고 판단하면 검색 도구를 쓰는 구조로 설계되어 있습니다. 또한 MCP를 통해 특정 데이터 소스, API, VectorDB, 전문 시스템과 연결할 수 있습니다.

이 차이는 중요합니다. 단순 첨부 방식은 “사용자가 어떤 자료가 필요한지 미리 알고 넣어주는 방식”입니다. 반면 MCP나 VectorDB 기반 연결은 “AI가 업무 중 필요한 정보를 찾아 쓰는 방식”으로 발전할 수 있습니다.

AX 관점에서는 후자가 더 중요합니다. AI가 실제 업무를 하려면 필요한 정보를 사람이 매번 챙겨주는 것이 아니라, AI가 안전한 경로로 필요한 시스템과 데이터에 접근할 수 있어야 합니다.

### 6.3 산출물 중심 업무 흐름의 차이

P-GPT는 답변을 잘 생성해도 결과가 대화창 안에 머무르는 경우가 많습니다. HTML 보고서를 만들어도 코드 원문이 출력될 뿐, 실제 화면으로 렌더링해 검토하기 어렵습니다.

MyHarness는 산출물을 파일로 만들고, 저장하고, 미리보고, 다시 수정하는 흐름을 지원합니다. HTML 보고서나 대시보드는 오른쪽 preview panel에서 바로 확인할 수 있고, 사용자는 “표를 추가해줘”, “색상을 줄여줘”, “요약을 앞에 배치해줘”처럼 이어서 개선할 수 있습니다.

업무에서는 이 차이가 큽니다. 최종적으로 남는 것은 채팅 답변이 아니라 문서, 보고서, 표, 대시보드, 분석 파일 같은 산출물이기 때문입니다.

### 6.4 결론: P-GPT는 채팅형 AI, MyHarness는 업무 실행형 AI 환경

P-GPT는 우리 회사가 AI를 쓸 수 있게 해주는 중요한 지능 기반입니다. 그러나 AX를 위해서는 그 지능을 업무 프로세스와 사내 정보, 산출물 관리, 반복 업무 자산화에 연결해야 합니다.

MyHarness는 바로 그 연결을 담당합니다.

```text
P-GPT = 사내에서 사용할 수 있는 채팅형 AI
MyHarness = 지능을 업무 수행으로 바꾸는 Agent 업무 환경
```

따라서 MyHarness의 메시지는 “채팅 답변을 더 잘 받자”에 머무르지 않습니다. 핵심은 “사내 LLM을 실제 업무에 붙이려면 Skill, MCP, Harness가 필요하고, 그 결과 사용자는 기존 채팅형 AI보다 개선된 업무용 AI 도구를 사용할 수 있다”입니다.

## 7. 핵심 기능

### 7.0 UI/UX는 Harness의 핵심 표면이다

UI/UX는 Harness와 별개의 장식 요소가 아니라, Harness가 사용자에게 드러나는 가장 중요한 표면입니다.

Harness는 LLM, Skill, MCP를 감싸는 실행 환경입니다. 그런데 사용자가 그 실행 환경을 실제로 믿고 쓰려면 좋은 UI/UX가 필요합니다.

- 어떤 작업이 진행 중인지 보여줘야 합니다.
- AI가 어떤 도구와 자료를 사용했는지 확인할 수 있어야 합니다.
- AI가 도구 실행 결과를 다시 보고 다음 작업으로 이어가는 흐름이 보여야 합니다.
- 결과물이 대화창에 묻히지 않고 파일과 preview로 이어져야 합니다.
- 사용자가 “수정해줘”, “다시 만들어줘”, “저장해줘”, “공유해줘”를 자연스럽게 할 수 있어야 합니다.
- 복잡한 Agent 기능을 일반 사용자도 부담 없이 사용할 수 있어야 합니다.

따라서 UI/UX는 Harness의 부가 기능이 아니라, AI Agent를 실제 업무 도구로 바꾸는 핵심 요소입니다. 기술적으로 아무리 강한 Agent라도 화면 경험이 불편하면 조직 안에서 확산되기 어렵습니다.

### 7.1 브라우저 기반 Agent 작업 화면

사용자는 별도 개발 환경이나 복잡한 명령어 없이 브라우저에서 Agent를 사용할 수 있습니다.

화면은 크게 대화 영역, 작업 진행 영역, 산출물 미리보기 영역으로 구성됩니다. 사용자는 Agent에게 자연어로 지시하고, Agent가 어떤 단계를 거쳐 일하는지 확인하며, 생성된 파일을 바로 열어볼 수 있습니다.

### 7.2 워크스페이스 기반 업무 관리

MyHarness는 작업을 프로젝트별 워크스페이스로 나눕니다.

예를 들어 시장 조사, 보고서 작성, 데이터 분석, 코드 수정 같은 업무를 각각 별도 폴더에서 진행할 수 있습니다. Agent가 만든 파일도 해당 워크스페이스에 저장되므로, 작업 결과를 나중에 다시 찾기 쉽습니다.

### 7.3 투명한 작업 과정

AI Agent를 업무에 쓰기 위해서는 결과뿐 아니라 과정도 중요합니다.

MyHarness는 Agent가 어떤 도구를 사용했는지, 어떤 파일을 읽었는지, 어떤 작업이 진행 중인지 보여줍니다. 이는 사용자가 결과를 더 쉽게 검토하고, 오류가 있을 때 원인을 추적하는 데 도움이 됩니다.

### 7.4 산출물 미리보기

Agent가 만든 결과물은 대화창에 묻히지 않고 파일 카드와 미리보기 패널로 연결됩니다.

지원하는 산출물 예시는 다음과 같습니다.

- HTML 보고서와 대시보드
- Markdown 문서
- JSON, CSV, TXT 파일
- 이미지
- PDF
- Office 계열 파일

특히 HTML 보고서나 대시보드는 브라우저에서 바로 확인할 수 있어, “AI가 만든 결과물을 즉시 검토하고 수정 요청하는” 흐름이 자연스럽습니다.

### 7.5 스킬, MCP, 플러그인 확장

MyHarness는 사내 업무에 맞는 기능을 계속 추가할 수 있도록 설계되어 있습니다.

- 스킬: 특정 업무 방식이나 산출물 품질 기준을 Agent에게 알려주는 지침
- MCP: 외부 데이터, 내부 시스템, 전문 도구와 연결하는 인터페이스
- 플러그인: 기능을 묶어서 배포하고 관리하는 확장 단위

이 구조 덕분에 MyHarness는 범용 AI 도구에 머무르지 않고, 회사 업무 환경에 맞는 Agent 플랫폼으로 확장될 수 있습니다. 중요한 것은 이 확장이 단순 기능 추가가 아니라, 회사의 업무 지식을 AI가 사용할 수 있는 형태로 축적하는 과정이라는 점입니다.

### 7.6 사내 배포와 운영을 고려한 구조

MyHarness는 Windows 환경에서 쉽게 실행할 수 있도록 설치기와 실행 파일을 제공합니다.

설정과 인증 정보는 프로젝트 내부의 `.myharness` 폴더를 중심으로 관리됩니다. 따라서 폴더를 복사하거나 다른 PC로 옮겨도 비교적 일관된 실행 환경을 유지할 수 있습니다.

## 8. 사용 시나리오

### 시나리오 1: 보고서 작성

사용자가 “철강 시장 동향을 조사해서 HTML 보고서로 정리해줘”라고 요청합니다.

MyHarness의 Agent는 자료를 찾고, 핵심 내용을 정리하고, 표와 시각 요소가 포함된 HTML 보고서를 생성합니다. 사용자는 오른쪽 미리보기 패널에서 결과를 바로 확인하고, “요약을 더 짧게”, “표를 추가해줘”, “임원 보고용 톤으로 바꿔줘”처럼 이어서 수정할 수 있습니다.

### 시나리오 2: 사내 데이터 연결

MCP를 통해 특정 데이터 소스나 업무 시스템이 연결되어 있다면, Agent는 그 데이터를 조회해 분석할 수 있습니다.

예를 들어 경제 지표, 산업 통계, 법령 정보, 사내 지식베이스 같은 자료를 연결해 업무 맥락에 맞는 답변과 산출물을 만들 수 있습니다.

### 시나리오 3: 프로젝트 파일 기반 업무

사용자는 현재 워크스페이스의 파일을 `@` 참조로 지정할 수 있습니다.

Agent는 해당 파일을 읽고 요약하거나, 비교하거나, 필요한 형식으로 변환할 수 있습니다. 따라서 단순히 “AI에게 물어보는” 수준을 넘어, 실제 업무 파일을 중심으로 작업할 수 있습니다.

### 시나리오 4: 반복 업무의 스킬화

자주 수행하는 보고서 형식, 검토 기준, 조사 절차는 스킬로 만들 수 있습니다.

이렇게 하면 사용자가 매번 긴 지침을 입력하지 않아도, 조직에서 합의한 방식으로 일관된 결과물을 만들 수 있습니다.

## 9. AX 관점에서의 의의

MyHarness의 의의는 단순히 편리한 AI 화면을 만드는 데 있지 않습니다. 더 큰 의미는 AX를 실제 업무 현장에 적용하기 위한 기반을 만든다는 점입니다.

### 9.1 AI Agent 접근성 향상

Agent 기술은 강력하지만, CLI나 개발자 도구에 익숙하지 않은 사용자에게는 진입 장벽이 높습니다.

MyHarness는 이를 브라우저 UI로 낮춥니다. 사용자는 복잡한 명령어를 몰라도 대화, 파일 참조, 미리보기, 설정 화면을 통해 Agent를 사용할 수 있습니다.

### 9.2 신뢰 가능한 업무 흐름 제공

AI 결과를 업무에 적용하려면 “어떻게 이 결과가 나왔는지”를 확인할 수 있어야 합니다.

MyHarness는 작업 과정, 도구 사용, 파일 참조, 산출물 생성을 화면에 드러냅니다. 이는 Agent 결과를 검토하고 조직 안에서 설명할 수 있게 만드는 중요한 장치입니다.

### 9.3 사내 Agent 플랫폼의 기반

MyHarness는 특정 모델 하나를 쓰는 단일 챗봇이 아닙니다.

Provider, 스킬, MCP, 플러그인, 워크스페이스, 산출물 미리보기 구조를 갖고 있기 때문에, 사내 업무에 맞춰 계속 확장할 수 있는 기반 플랫폼에 가깝습니다.

### 9.4 업무 지식의 자산화

기존에는 업무 노하우가 개인의 경험, 문서, 메신저 대화, 암묵지 형태로 흩어져 있었습니다.

MyHarness에서 스킬을 만든다는 것은 이런 노하우를 AI가 읽고 실행할 수 있는 형태로 바꾸는 일입니다. 보고서 작성 기준, 검토 체크리스트, 데이터 해석 방식, 승인 문서 형식, 자주 발생하는 오류 대응법이 스킬로 축적되면, 개인의 경험이 조직의 재사용 가능한 자산이 됩니다.

이 작업은 현업 직원의 참여 없이는 제대로 되기 어렵습니다. DX 전문부서는 Skill을 만들 수 있는 구조와 가이드, 테스트 환경을 제공할 수 있지만, 업무의 실제 판단 기준은 각 부서와 담당자에게 있습니다. 따라서 Skill 제작은 “현업 지식의 문서화”이자 “AI가 읽을 수 있는 업무 표준 만들기”에 가깝습니다.

좋은 Skill은 다음 cycle을 거쳐 만들어집니다.

```text
업무 지식 문서화 → MyHarness에서 사용 → 결과 점검 → 테스트 → Skill 수정 → 재사용
```

이 cycle이 돌아가야 Skill은 단순 설명서가 아니라 실제 업무에 쓸 수 있는 조직 자산이 됩니다. MyHarness는 AI Agent 사용 도구이면서 동시에 이 Skill 제작·검증 cycle을 운영하기 위한 플랫폼입니다.

### 9.5 사내 정보 접근의 표준화

AI가 업무를 하려면 사내 정보를 안전하게 읽을 수 있어야 합니다. 이 역할을 MCP가 맡습니다.

MCP는 AI와 사내 데이터, 시스템, 문서 저장소, 전문 API 사이의 표준화된 연결 통로입니다. MCP가 잘 구축될수록 AI는 더 많은 업무 맥락을 이해하고, 더 정확한 결과를 만들 수 있습니다.

향후에는 온톨로지 같은 지식 구조도 함께 연결될 수 있습니다. 이 경우 AI는 단순히 문서 안의 단어를 찾는 것을 넘어, 업무 개념 간 관계와 맥락을 활용하는 방향으로 확장될 수 있습니다. 다만 발표에서는 이를 핵심 주제가 아니라 “추가 확장 방향”으로 가볍게 언급하는 정도가 적절합니다.

### 9.6 업무방식의 변화

AX는 도구 도입만으로 완성되지 않습니다. 업무 방식도 함께 바뀌어야 합니다.

앞으로의 업무는 사람이 모든 단계를 직접 수행하는 방식에서, 사람이 목표와 기준을 제시하고 AI가 초안을 만들며, 사람이 검토하고 보완하는 방식으로 이동할 가능성이 큽니다.

즉, 업무자는 단순 실행자가 아니라 다음 역할을 더 많이 하게 됩니다.

- AI에게 명확한 목표와 제약을 주는 지시자
- 결과물의 품질과 사실성을 판단하는 검토자
- 반복 업무를 스킬로 정리하는 업무 설계자
- 사내 데이터와 시스템을 MCP로 연결하는 개선 제안자
- AI가 만든 결과를 실제 의사결정과 실행으로 연결하는 책임자

MyHarness는 이런 업무방식 전환을 실험하고 정착시키기 위한 공간입니다.

## 10. 우리가 집중해야 할 것

발표에서 가장 강조할 메시지는 다음과 같습니다.

1. 충분히 똑똑한 LLM만으로는 AX가 완성되지 않습니다.
2. AI가 업무를 하려면 업무 도메인 지식, 사내 정보 접근, 실행 환경이 함께 필요합니다.
3. 스킬은 회사의 업무 지식을 AI가 사용할 수 있는 형태로 바꾸는 작업입니다.
4. MCP는 AI가 사내 데이터와 시스템에 안전하게 접근하게 하는 연결 통로입니다.
5. 미래에 더 좋은 LLM이 나오더라도 우리 회사 업무에 맞는 Skill과 MCP는 계속 필요합니다.
6. Harness는 LLM, 스킬, MCP를 둘러싸 실제 사용자가 쓸 수 있는 업무 환경으로 만들어주는 기반입니다.
7. Harness는 AI가 도구를 쓰고 결과를 다시 확인하는 closed-loop Agentic Workflow를 가능하게 해야 합니다.
8. UI/UX는 Harness의 핵심 표면입니다. AI가 무엇을 하고 있는지 보이고, 결과물을 바로 검토하고 수정할 수 있어야 실제 업무 도구가 됩니다.
9. MCP 위에는 VectorDB, 업무 시스템, 전문 API뿐 아니라 향후 온톨로지 같은 지식 구조도 연결될 수 있습니다.
10. MCP 개발은 DX 관련 전문부서가 중심이 될 수 있지만, Skill 개발은 현업 직원의 참여가 필요합니다.
11. Skill은 만들고, 점검하고, 테스트하고, 수정하는 반복 cycle을 통해 성숙합니다.
12. 우리 회사가 AX를 제대로 도입하려면 스킬 개발, MCP 개발, Harness 고도화에 집중해야 합니다.
13. 동시에 업무 방식도 “사람이 직접 처리”에서 “사람이 설계하고 AI가 수행하며 사람이 검토”하는 방식으로 바뀌어야 합니다.

## 11. 실제 추진 사례: MyHarness 자체가 Agentic Coding 사례

MyHarness는 단순히 AI Agent 도입을 주장하기 위해 만든 개념 자료가 아닙니다. MyHarness 자체가 Agentic Coding을 통한 자력 개발 사례입니다.

2026년 4월 26일부터 혼자서 기획, 설계, 제작을 동시에 진행했고, 3일 만에 초기 프로토타입이 나왔습니다. 이후 약 2주간의 집중 개발을 거쳐 단순 PoC보다 높은 완성도의 현업 검증용 MVP/파일럿 수준으로 끌어올렸고, 경영기획 실장님과 본부장님께 보고했습니다. 1주일 뒤에는 사장님 보고까지 이어졌습니다.

일반적인 정식 과제 개발 방식이었다면 이 속도는 사실상 불가능했을 가능성이 큽니다. 비교 대상은 완전한 정식 운영 시스템이 아니라, 현업 보고와 검증이 가능한 MVP/파일럿 수준입니다. 이 정도의 초기 검증 가능한 결과물을 만들기 위해서도 보통은 다음 절차만으로 수개월이 걸릴 수 있습니다.

```text
문제 정의
→ 요구사항 정리
→ 현업 협의
→ 과제화 검토
→ 기본 설계
→ 업무량·일정 산정
→ 예산 검토
→ 투자/구매 심의
→ 업체 선정·계약
→ 상세 설계
→ 개발
→ 테스트·검수
→ 배포
```

물론 이런 절차가 불필요하다는 뜻은 아닙니다. 다만 Agentic Coding을 활용하면 정식 과제화 전에 초기 아이디어를 실제로 만져보고 현업 검증까지 할 수 있는 MVP/파일럿을 만드는 속도가 극적으로 빨라질 수 있습니다. MyHarness는 “AI로 업무를 바꾼다”는 말 이전에, AI와 함께 개발하는 방식이 이미 업무 속도를 바꿀 수 있음을 보여주는 사례입니다.

다만 현재 수준을 정식 운영 시스템으로 과장해서 볼 필요는 없습니다. 보안, 일부 버그와 안정성, 다수 사용자 활용 최적화, 운영 정책 등은 정식 확대 전에 추가 보완이 필요한 영역입니다. 핵심은 “완성품을 며칠 만에 만들었다”가 아니라, “현업이 보고 판단할 수 있는 수준의 결과물을 매우 빠르게 만들었다”는 점입니다.

## 12. 다음 과제: 영업이익 변동 원인 추적 Agent

사장님 보고 이후에는 “영업이익 변동 원인 추적 Agent를 구현하라”는 과제를 받았습니다.

이를 위해 올해 안에 다음 방향으로 구현을 시도할 예정입니다.

- 사내 원가 DB(DataLake)를 업무 개념과 관계 중심으로 온톨로지화
- 자연어로 필요한 데이터를 추출할 수 있는 MCP 연결
- 원가분석 관련 도메인 지식을 Skill로 정리
- MyHarness 안에서 원인 추적, 데이터 조회, 분석 메모, 보고서 초안 생성까지 이어지는 Agent 흐름 구현

이 과제는 앞에서 설명한 구조가 실제 경영 이슈에 적용되는 사례입니다.

```text
LLM: 원인 추론과 설명
Skill: 원가분석 도메인 지식
MCP: DataLake와 자연어 데이터 추출
Ontology: 원가 항목과 업무 개념 간 관계 구조
Harness: 분석 과정, 결과물, 검토 흐름을 감싸는 업무 환경
```

온톨로지는 여기서 핵심 주인공이라기보다, MCP와 함께 AI가 사내 데이터를 더 정확한 업무 맥락으로 이해하도록 돕는 지식 구조입니다. 발표에서는 특정 회사명은 언급하지 않고, “향후 사내 원가 데이터를 온톨로지화해 연결할 예정” 정도로 전달하는 것이 적절합니다.

## 13. 5분 발표 구성안

### 1분: 문제 제기

AI Agent는 강력하지만 최신 LLM만 도입한다고 업무가 자동화되지는 않습니다. 실제 업무를 하려면 회사의 업무 지식, 사내 정보 접근, 실행 환경이 함께 필요합니다.

### 1분: AI 업무 수행 공식

LLM은 지능을 제공하고, 스킬은 업무 도메인 지식을 제공하며, MCP는 사내 정보 접근을 제공합니다. Harness는 이 세 요소를 둘러싸 하나의 사용 가능한 업무 환경으로 만듭니다. 이 구조가 있어야 AI가 실제 업무를 수행할 수 있습니다.

### 1분: P-GPT와 MyHarness의 차이

P-GPT는 사내 LLM 지능을 제공하지만, MyHarness는 그 지능을 업무 수행 환경으로 확장합니다. 스킬, MCP, 산출물 preview, 파일 저장과 수정 흐름이 결합되어야 AI가 실제 업무에 붙을 수 있습니다.

### 1분: MyHarness 소개와 활용 사례

MyHarness는 이 구조를 구현하기 위한 브라우저 기반 업무 지원 시스템입니다. 사용자는 Agent에게 업무를 지시하고, 작업 과정을 확인하고, 생성된 산출물을 바로 검토할 수 있습니다.

보고서 작성, 자료 조사, 데이터 분석, 문서 요약, 반복 업무 스킬화 등에 사용할 수 있습니다. 또한 MyHarness 자체가 Agentic Coding으로 빠르게 만들어진 사례이며, 다음 단계로 영업이익 변동 원인 추적 Agent 구현을 추진할 예정입니다.

### 1분: 결론

AX 도입의 핵심은 모델 하나를 사오는 것이 아니라, 우리 업무에 맞는 스킬, 사내 시스템과 연결되는 MCP, 그리고 이를 안전하고 편리하게 감싸는 Harness를 함께 개발하는 것입니다.

## 14. 발표용 짧은 스크립트

안녕하세요. 오늘 소개드릴 MyHarness는 사내 구성원이 AI Agent를 더 쉽게 사용하고, 실제 업무에 적용할 수 있도록 만든 브라우저 기반 업무 지원 시스템입니다.

요즘 AI는 단순히 질문에 답하는 수준을 넘어, 파일을 읽고, 자료를 찾고, 보고서를 만들고, 여러 단계의 작업을 수행하는 Agent 형태로 발전하고 있습니다. 그런데 여기서 중요한 점은, 충분히 똑똑한 모델 하나만으로는 회사 업무가 자동화되지 않는다는 것입니다.

AI가 실제 업무를 하려면 먼저 세 가지 재료가 필요합니다. 첫째, 판단과 추론을 담당하는 충분히 똑똑한 LLM이 필요합니다. 둘째, 우리 회사의 업무 절차, 보고서 형식, 검토 기준 같은 도메인 지식이 스킬 형태로 제공되어야 합니다. 셋째, AI가 사내 데이터와 시스템에 접근할 수 있도록 MCP 같은 인터페이스가 필요합니다. 그리고 이 세 가지를 사용자가 안전하고 편리하게 쓸 수 있도록 감싸는 Harness가 필요합니다.

아무리 머리가 좋은 사람도 우리 회사에 처음 들어오면 바로 일을 할 수 없습니다. 회사 업무의 용어, 기준, 예외, 보고 방식은 따로 배워야 합니다. LLM도 마찬가지입니다. 앞으로 모델이 더 좋아져도 우리 회사 업무에 맞는 Skill과 사내 시스템을 연결하는 MCP는 계속 필요합니다.

Harness가 중요한 이유는 단순히 화면을 제공하기 때문만은 아닙니다. Harness는 AI가 업무 중 필요한 도구를 쓰고, 그 결과를 다시 보고, 부족하면 다음 행동을 이어가는 closed loop를 만듭니다. 이 순환이 있어야 AI는 “긴 답변을 잘하는 챗봇”이 아니라 “도구를 쓰며 업무를 수행하는 Agent”에 가까워집니다.

우리 회사에는 이미 P-GPT가 있기 때문에, AI를 사용할 수 있는 기본 기반은 갖춰져 있습니다. 다만 P-GPT는 주로 질문을 입력하고 답변을 받는 채팅형 AI에 가깝습니다. “나의 GPT”를 통해 커스텀 프롬프트를 저장할 수는 있지만, MyHarness의 스킬은 그보다 더 넓습니다. 상황에 맞게 선택·추천·호출되도록 설계할 수 있고, 단순 프롬프트뿐 아니라 스크립트, 참고 파일, 템플릿, routing과 연결될 수 있습니다.

또한 P-GPT에서는 인터넷 검색을 사용자가 별도로 선택해야 하는 경우가 많고, ECM 연동도 기본적으로 자료를 첨부해 읽히는 방식에 가깝습니다. MyHarness는 Agent가 필요하다고 판단하면 검색 도구를 쓰는 구조로 설계되어 있고, MCP를 통해 VectorDB나 사내 시스템, 전문 API와 연결되는 방향으로 확장할 수 있습니다. 향후 온톨로지 같은 지식 구조가 연결되면, AI가 업무 개념 간 관계까지 활용하는 방향도 가능해집니다.

산출물 측면에서도 차이가 있습니다. P-GPT는 결과를 대화창에 출력하는 방식이 중심이라 HTML을 만들어도 원문 코드가 출력되는 데 그칠 수 있습니다. MyHarness는 파일을 만들고, 저장하고, preview로 렌더링하고, 다시 수정하는 흐름을 지원합니다. 즉, 결과가 채팅 답변에 머무르지 않고 실제 업무 산출물이 됩니다.

MyHarness는 바로 이 Harness 역할을 하는 시스템입니다. 사용자는 브라우저에서 자연어로 지시하고, Agent가 어떤 작업을 수행하는지 화면에서 확인할 수 있습니다. Agent가 만든 HTML 보고서, Markdown 문서, 이미지, PDF 같은 산출물은 오른쪽 패널에서 바로 미리볼 수 있습니다.

또한 MyHarness는 워크스페이스 단위로 프로젝트와 기록을 관리합니다. 사용자는 현재 작업 폴더의 파일을 참조해 Agent에게 분석이나 변환을 요청할 수 있고, 자주 쓰는 업무 방식은 스킬로 만들어 반복 활용할 수 있습니다. MCP를 통해 외부 데이터나 사내 시스템과 연결하는 확장도 가능합니다.

중요한 것은 MyHarness 자체가 이런 변화를 보여주는 실제 사례라는 점입니다. 4월 26일부터 혼자서 기획, 설계, 제작을 동시에 시작했고, 3일 만에 초기 프로토타입이 나왔습니다. 이후 2주간 집중 개발을 거쳐 단순 PoC보다 높은 현업 검증용 MVP/파일럿 수준으로 만들었고, 경영기획 실장님과 본부장님 보고까지 갔으며, 1주일 뒤 사장님 보고까지 이어졌습니다. 일반적인 정식 과제 방식이었다면 문제 정의, 요구사항 정리, 현업 협의, 과제화 검토, 기본 설계, 업무량·일정 산정, 예산 검토, 심의, 업체 선정과 계약을 거치는 데만 수개월이 걸렸을 가능성이 큽니다. Agentic Coding을 통한 자력 개발이 아니었다면 이 속도는 어려웠을 것입니다. 물론 정식 운영 확대를 위해서는 보안, 안정성, 버그 개선, 다수 사용자 최적화 같은 보완 과제가 남아 있습니다.

그리고 사장님 보고 이후에는 “영업이익 변동 원인 추적 Agent”를 구현하라는 과제를 받았습니다. 이를 위해 사내 원가 DB, 즉 DataLake를 온톨로지화하고, 자연어로 데이터를 추출할 수 있는 MCP를 연결하며, 원가분석 도메인 지식을 스킬화하는 방향으로 올해 안에 구현을 시도할 예정입니다. 이것은 앞에서 설명한 LLM, Skill, MCP, Harness 구조가 실제 경영 이슈에 적용되는 다음 단계입니다.

결국 MyHarness의 핵심 의의는 AI Agent를 전문가용 실험 도구가 아니라, 사내 구성원이 실제 업무에서 쓸 수 있는 생산성 도구로 바꾸는 데 있습니다. 그리고 더 크게 보면, AX 도입을 위해 우리 회사가 어디에 집중해야 하는지를 보여줍니다.

앞으로 중요한 것은 최신 모델을 쓰는 것만이 아닙니다. 우리 업무를 스킬로 정리하고, 사내 정보를 MCP로 연결하고, 이 모든 것을 Harness 안에서 안전하게 운영하는 것입니다. 동시에 업무 방식도 바뀌어야 합니다. 사람이 모든 단계를 직접 처리하는 방식에서, 사람이 목표와 기준을 제시하고 AI가 초안을 만들며, 사람이 검토하고 개선하는 방식으로 전환해야 합니다.

여기서 MCP와 Skill의 성격은 다릅니다. MCP는 사내 시스템과 데이터를 연결하는 전문 인프라이므로 DX 관련 전문부서가 중심이 될 수 있습니다. 반면 Skill은 현업 도메인 지식을 AI가 실행 가능한 형태로 문서화하는 일이므로 직원들의 참여가 반드시 필요합니다. 각 부서가 자기 업무의 기준과 노하우를 Skill로 만들고, MyHarness에서 써보고, 결과를 점검하고, 다시 수정하는 cycle이 돌아가야 합니다. 이것이 MyHarness를 만든 중요한 목적입니다.

여기서 UI/UX도 매우 중요합니다. UI/UX는 단순히 예쁘게 보이는 문제가 아니라, Harness가 실제 업무 도구로 받아들여지는 방식입니다. 사용자는 AI가 무엇을 하고 있는지 보고, 생성된 결과물을 바로 preview하고, 수정하고, 저장하고, 공유할 수 있어야 합니다. 이 경험이 좋아야 AI Agent가 실험실 안의 기술이 아니라 현장의 업무 방식으로 들어올 수 있습니다.

MyHarness는 이러한 변화를 실험하고 확산하기 위한 사내 Agent 플랫폼의 출발점입니다.

## 15. 한 장 슬라이드 요약

제목: MyHarness - 사내 AI Agent 업무 지원 시스템

핵심 설명:

- P-GPT는 사내 LLM 지능을 제공하고, MyHarness는 그 지능을 업무 수행 환경으로 확장합니다.
- LLM은 지능을 제공하고, 스킬은 업무 지식을 제공하며, MCP는 사내 정보 접근을 제공합니다.
- 아무리 좋은 LLM이 나오더라도 우리 회사 업무를 수행하려면 Skill과 MCP가 필요합니다.
- Harness는 LLM·Skill·MCP를 둘러싸 실제 사용자가 쓸 수 있는 업무 환경으로 만듭니다.
- Harness는 AI가 도구를 사용하고 결과를 다시 확인하는 closed-loop Agentic Workflow를 가능하게 합니다.
- MyHarness는 브라우저에서 AI Agent와 대화하며 실제 업무를 수행하게 하는 시스템입니다.
- 작업 과정, 도구 사용, 산출물 생성을 투명하게 표시합니다.
- 워크스페이스별 파일, 기록, 결과물 관리와 산출물 미리보기를 지원합니다.
- UI/UX는 Harness의 핵심 표면으로, 사용자가 AI 업무 과정을 이해하고 결과물을 수정·저장·공유하게 만듭니다.
- MyHarness는 직원들이 Skill을 만들고, 사용하고, 점검하고, 테스트하고, 수정하는 플랫폼입니다.
- MCP는 전문부서가 만들 수 있지만, Skill은 현업 도메인 지식을 가진 직원들이 함께 축적해야 합니다.

기대 효과:

- AI Agent 접근성 향상
- 업무 결과 검토와 설명 가능성 강화
- 반복 업무 자동화와 산출물 생성 효율화
- 사내 업무 지식의 스킬화
- 직원 참여형 Skill 제작 cycle 정착
- 사내 데이터와 시스템의 MCP 연결
- Agentic Coding을 통한 빠른 자력 개발과 실험
- 영업이익 변동 원인 추적 Agent 같은 실제 경영 과제 적용
- AX 도입을 위한 Agent 플랫폼 기반 확보

핵심 결론:

```text
AX = Harness 안에서 돌아가는 LLM·Skill·MCP
   + 사람의 목표 설정·검토·책임
   + AI와 함께 일하는 업무방식 전환
```
