# MyHarness 발표 최적화 슬라이드 구성안

작성일: 2026-06-11

## 구성 원칙

기존 20페이지 구성은 설명 요소가 충분하지만, 실제 발표에서는 메시지가 반복될 수 있습니다. 최적안은 12페이지로 줄이되, 각 장표는 혼자 읽어도 의미가 전달되도록 본문 문구를 충분히 담습니다.

핵심 흐름은 다음과 같습니다.

```text
이미 P-GPT가 있다
→ 하지만 채팅형 AI와 업무 수행 Agent는 다르다
→ 업무 수행에는 LLM·Skill·MCP를 감싸는 Harness가 필요하다
→ Harness는 AI가 도구를 쓰고 결과를 다시 보는 closed loop를 만든다
→ MCP는 전문부서가 만들 수 있지만, Skill은 현업 직원이 함께 만들어야 한다
→ MyHarness는 그 구조를 실험한 현업 검증용 MVP/파일럿이다
→ Agentic Coding으로 비정상적으로 빠르게 만들었다
→ 다음 과제는 영업이익 변동 원인 추적 Agent다
→ AX를 위해 Skill, MCP, Harness, 업무방식 전환에 집중해야 한다
```

## Slide 1. 제목

### 제목

MyHarness: AI를 채팅창에서 업무 현장으로

### 슬라이드 본문

MyHarness는 사내 LLM을 단순 질의응답 도구로 쓰는 것을 넘어, 실제 업무 파일과 데이터, 도구, 산출물을 다루는 Agent 업무 환경으로 확장하기 위한 시도입니다.

핵심 변화는 다음 한 문장으로 요약됩니다.

```text
답변을 받는 AI에서, 업무 결과물을 만드는 AI로
```

### 화면 표현

- 흰 배경, 손그림 스케치
- 왼쪽: 채팅창 안의 AI
- 오른쪽: 보고서, 데이터, 파일을 들고 움직이는 Agent
- 중앙 화살표: `답변 생성 → 업무 수행`

### 발표 포인트

처음부터 “프로그램 기능 소개”가 아니라 “AX를 위해 어떤 업무 실행 체계가 필요한가”를 설명하는 발표라고 방향을 잡습니다.

## Slide 2. 문제 제기

### 제목

이미 P-GPT가 있는데, 왜 또 MyHarness인가?

### 슬라이드 본문

P-GPT 덕분에 우리 회사는 이미 안전한 환경에서 AI를 사용할 수 있습니다. 이 기반은 매우 중요합니다. 다만 P-GPT는 기본적으로 사용자가 질문하고 AI가 답변을 출력하는 채팅형 AI에 가깝습니다.

실제 업무에서는 답변만으로 끝나지 않습니다.

- 자료를 찾고
- 사내 문서와 데이터를 확인하고
- 결과물을 파일로 만들고
- 화면에서 검토하고
- 다시 수정하고
- 저장하거나 공유해야 합니다.

따라서 질문은 “AI가 있느냐”가 아니라, “AI가 업무 흐름 안으로 들어와 있느냐”입니다.

### 화면 표현

- P-GPT 채팅창을 보는 사용자
- 옆에 쌓인 업무 파일, 보고서, 시스템 화면
- 말풍선: “답변은 받았는데, 이제 이걸 업무 결과물로 누가 만들지?”

### 발표 포인트

P-GPT를 부정하지 않습니다. 사내 LLM 기반이 이미 있다는 점을 출발점으로 인정하고, 그 지능을 업무 실행으로 연결하는 레이어가 필요하다고 전개합니다.

---

## Slide 3. 채팅형 AI의 한계

### 제목

채팅창 안의 AI는 똑똑하지만, 손발이 짧다

### 슬라이드 본문

채팅형 AI는 다음과 같은 일에는 강합니다.

- 질문에 답하기
- 문장 요약하기
- 초안 작성하기
- 아이디어 제안하기
- 간단한 분석 방향 잡기

하지만 업무 실행 관점에서는 한계가 있습니다.

- 결과가 대화창 안에 머무릅니다.
- HTML이나 대시보드도 실제 화면이 아니라 코드로 출력됩니다.
- 생성된 파일을 체계적으로 저장하고 관리하기 어렵습니다.
- 사내 시스템, 데이터, 업무 도구와의 연결이 제한적입니다.
- AI가 무엇을 했는지 작업 과정이 충분히 보이지 않습니다.

즉, 지능은 있지만 업무 환경은 부족합니다.

### 화면 표현

- 채팅창 안에 갇힌 똑똑한 AI
- 창 밖에 문서, DB, 대시보드, 결재 자료
- 문구: `지능은 있음 / 업무 환경은 부족`

## Slide 4. 핵심 구조

### 제목

AI가 업무를 하려면: Harness 안의 LLM·Skill·MCP

### 슬라이드 본문

AI가 실제 업무를 수행하려면 세 가지 핵심 재료가 필요합니다.

| 요소 | 의미 | 역할 |
|---|---|---|
| LLM | 충분히 똑똑한 지능 | 판단, 추론, 작성, 계획 |
| Skill | 업무 도메인 지식 | 회사 방식, 보고서 기준, 절차, 노하우 |
| MCP | 정보 접근 인터페이스 | 사내 데이터, 시스템, API, 지식 저장소 연결 |

하지만 이 세 가지가 흩어져 있으면 사용자는 여전히 복사, 붙여넣기, 저장, 수정, 공유를 직접 해야 합니다.

그래서 Harness가 필요합니다.

```text
Harness = LLM·Skill·MCP를 감싸 실제 업무 환경으로 만드는 실행 레이어
```

Harness의 또 다른 핵심 역할은 Agentic Workflow를 가능하게 하는 것입니다. 즉, AI가 한 번 답변하고 끝나는 것이 아니라 스스로 작업 결과를 보고 다음 행동을 결정하는 closed loop를 만듭니다.

```text
Observe → Plan → Tool-use → Result → Reflect/Revise
```

여기서 Tool-use가 매우 중요합니다. 검색, 파일 읽기·쓰기, preview, 코드 실행, MCP 조회 같은 도구를 AI가 상황에 맞게 사용하고, 그 결과를 다시 확인할 수 있어야 Agent가 실제 업무를 수행할 수 있습니다.

UI/UX는 이 Harness가 사용자에게 드러나는 핵심 표면입니다. 사용자가 AI의 작업을 보고, 결과물을 검토하고, 수정하고, 저장할 수 있어야 합니다.

### 화면 표현

- 큰 외곽선: `Harness`
- Harness 안에 세 카드: `LLM`, `Skill`, `MCP`
- Harness 내부 순환 화살표: `Observe → Plan → Tool-use → Result → Revise`
- 외곽선 표면에 `UI/UX`, `작업 기록`, `권한`, `파일`, `Preview`, `공유`, `도구 실행 결과`

### 발표 포인트

덧셈 공식처럼 보이지 않게 합니다. Harness는 네 번째 부품이 아니라 세 요소를 업무 환경으로 감싸는 실행 레이어입니다.

## Slide 5. Skill의 의미

### 제목

Skill은 프롬프트가 아니라 업무 지식의 자산화

### 슬라이드 본문

P-GPT의 “나의 GPT”는 커스텀 프롬프트를 저장해 특정 방식으로 답변하도록 유도할 수 있습니다. 이 기능은 유용하지만, 기본적으로는 개인화된 지침에 가깝습니다.

MyHarness의 Skill은 더 넓은 개념입니다.

Skill이 필요한 이유는 단순합니다. 아무리 머리가 좋은 사람도 우리 회사에 처음 오면 바로 일을 할 수 없습니다. 내부 용어, 보고 방식, 판단 기준, 예외 처리, 검토 포인트를 모르기 때문입니다. LLM도 마찬가지입니다. 모델의 지능이 아무리 좋아져도 우리 회사 업무를 하려면 회사 업무 지식이 필요하고, 그 지식을 AI가 실행 가능한 형태로 정리한 것이 Skill입니다.

- 업무 절차와 판단 기준을 담을 수 있습니다.
- 보고서 형식, 검토 체크리스트, 산출물 품질 기준을 정리할 수 있습니다.
- 필요하면 스크립트, 참고 파일, 템플릿, 예시 자료와 연결할 수 있습니다.
- 상황에 맞춰 선택·추천·호출되도록 설계할 수 있습니다.
- routing을 통해 특정 업무를 특정 Skill이나 MCP 흐름으로 보낼 수 있습니다.

즉, Skill은 개인 프롬프트가 아니라 조직의 업무 노하우를 AI가 실행 가능한 형태로 바꾸는 자산입니다.

여기서 중요한 차이가 있습니다. MCP는 DX 관련 전문부서가 표준 인터페이스와 보안 체계에 맞춰 만들 수 있습니다. 하지만 Skill은 실제 업무를 아는 사람들이 참여해야 합니다. 원가분석, 기획, 구매, 영업, 품질, 설비, 인사 같은 각 업무의 판단 기준과 암묵지는 현업 안에 있기 때문입니다.

Skill은 한 번 써놓고 끝나는 문서가 아닙니다.

```text
만들기 → 사용해보기 → 점검하기 → 테스트하기 → 수정하기
```

이 반복 cycle을 통해 좋은 Skill이 만들어집니다. 그래서 회사에는 Skill을 만들고, 써보고, 고치고, 공유할 수 있는 플랫폼이 필요합니다. MyHarness를 만든 핵심 목적 중 하나가 바로 이 지점입니다.

### 화면 표현

- 왼쪽: “나의 GPT” 메모장
- 오른쪽: Skill 도구상자
- 도구상자 안: `SKILL.md`, `.py`, template, reference, routing
- 아래쪽 순환 화살표: `작성 → 사용 → 점검 → 테스트 → 수정`
- 현업 직원들이 Skill 책장을 함께 채우는 스케치
- 문구: `개인 프롬프트 → 조직 업무 자산`

## Slide 6. MCP와 지식 연결

### 제목

MCP는 AI가 사내 정보에 접근하는 통로

### 슬라이드 본문

AI가 실제 업무를 하려면 필요한 자료를 사람이 매번 첨부해주는 방식만으로는 부족합니다. 업무 중 필요한 데이터를 안전한 경로로 조회하고, 시스템과 연결하고, 근거를 확인할 수 있어야 합니다.

MCP는 이 연결을 위한 인터페이스입니다.

연결 대상은 다음과 같이 확장될 수 있습니다.

- ECM과 사내 문서
- DataLake와 업무 데이터
- VectorDB 기반 지식 검색
- 사내 업무 시스템과 API
- 외부 통계, 법령, 산업 데이터
- 향후 온톨로지 기반 업무 개념 관계

온톨로지는 이 발표의 중심 주제는 아닙니다. 다만 향후 원가 DB 같은 사내 데이터를 업무 개념과 관계 중심으로 이해하기 위한 확장 방향으로 함께 언급합니다.

### 화면 표현

- AI Agent와 사내 시스템 사이에 `MCP 허브`
- 연결 대상: ECM, DataLake, VectorDB, API, 업무 시스템
- 작게: `Ontology / 개념 관계 지도`

## Slide 7. P-GPT vs MyHarness

### 제목

P-GPT는 채팅형 AI, MyHarness는 업무 실행형 AI 환경

### 슬라이드 본문

MyHarness는 P-GPT 같은 채팅형 AI를 업무 관점에서 한 단계 개선한 도구입니다. 차이는 “답변을 받는 AI”와 “업무 결과물을 만드는 AI”입니다.

| 구분 | P-GPT | MyHarness |
|---|---|---|
| 기본 성격 | AI 채팅 | Agent 업무 환경 |
| 업무 지식 | 나의 GPT, 커스텀 프롬프트 | Skill, routing, 파일·스크립트 연계 |
| 정보 접근 | 검색 클릭, 첨부 중심 | 검색·MCP 도구 사용 구조 |
| 작동 방식 | 질문 후 답변 | Observe→Tool-use→Result closed loop |
| 산출물 | 대화창 출력 중심 | 작성·저장·preview·수정·공유 |
| HTML/대시보드 | 코드 원문 출력 | 렌더링 preview와 수정 흐름 |
| 사용자 경험 | 답변 확인 | 업무 과정과 결과물 관리 |

요약하면, P-GPT는 “말은 잘하는 AI”이고 MyHarness는 “업무 결과물까지 만들게 하는 AI 환경”입니다.

### 화면 표현

- 2열 비교표
- 말풍선
- P-GPT: “말은 잘함”
- MyHarness: “일도 시킴”

## Slide 8. MyHarness 사용 경험

### 제목

대화, 작업 과정, 산출물이 한 화면에

### 슬라이드 본문

좋은 UI/UX는 Harness의 핵심입니다. AI Agent가 실제 업무 도구가 되려면 사용자가 다음을 한 화면에서 자연스럽게 이해할 수 있어야 합니다.

- 지금 어떤 작업이 진행 중인지
- AI가 어떤 도구를 사용했는지
- 도구 실행 결과를 AI가 다시 보고 다음 행동으로 이어가는지
- 어떤 파일과 자료를 참고했는지
- 어떤 산출물이 생성되었는지
- 결과물을 바로 열어보고 수정할 수 있는지

MyHarness는 대화창, 작업 과정, 워크스페이스, 도구 실행 결과, 산출물 preview를 하나의 흐름으로 묶습니다. 그래서 사용자는 “답변을 복사하는 사람”이 아니라 “AI와 함께 결과물을 다듬는 사람”이 됩니다. 동시에 AI도 자신이 수행한 작업의 결과를 다시 보고, 부족하면 검색하거나 파일을 고치거나 산출물을 재생성하는 다음 행동으로 이어갈 수 있습니다.

### 화면 표현

- MyHarness 화면 와이어프레임
- 왼쪽: 워크스페이스/히스토리
- 가운데: 대화
- 오른쪽: 산출물 preview
- 하단/측면: workflow timeline
- 말풍선: “AI가 뭘 하는지 보여야 믿고 맡긴다”

## Slide 9. 실제 개발 사례

### 제목

Agentic Coding이 만든 개발 속도

### 슬라이드 본문

MyHarness 자체가 Agentic Coding을 통한 자력 개발 사례입니다.

진행 흐름은 다음과 같습니다.

```text
4/26 기획·설계·제작 시작
→ 3일 후 초기 Prototype
→ 2주 집중 개발
→ 현업 검증용 MVP/파일럿 수준
→ 경영기획 실장님·본부장님 보고
→ 1주일 뒤 사장님 보고
```

일반적인 정식 과제 방식이라면, 현업 검증 가능한 MVP를 만드는 데에도 다음 절차만으로 수개월이 걸릴 수 있습니다.

```text
문제 정의 → 요구사항 정리 → 현업 협의 → 과제화 검토
→ 기본 설계 → 업무량·일정 산정 → 예산 검토
→ 투자/구매 심의 → 업체 선정·계약
→ 상세 설계 → 개발 → 테스트·검수 → 배포
```

중요한 점은 “정식 절차가 필요 없다”가 아닙니다. Agentic Coding을 활용하면 정식 과제화 전에 현업이 보고 판단할 수 있는 수준의 결과물을 훨씬 빠르게 만들 수 있다는 것입니다.

현재 MyHarness는 단순 PoC보다는 높은 완성도이지만, 정식 운영 전에는 보안, 안정성, 일부 버그, 다수 사용자 최적화, 운영 정책 보완이 필요합니다.

### 화면 표현

- 위쪽: MyHarness 개발 타임라인
- 아래쪽: 일반 정식 과제 절차가 길게 이어지는 낙서
- 말풍선: “정식 운영 전 보완은 필요, 하지만 초기 검증 속도는 완전히 달라짐”

### 슬라이드 본문

사장님 보고 이후 다음 과제를 받았습니다.

```text
영업이익 변동 원인 추적 Agent 구현
```

이 과제는 MyHarness가 지향하는 구조를 실제 경영 이슈에 적용하는 사례입니다.

## Slide 11. 업무방식 변화

### 제목

사람이 다 하는 업무에서, AI와 함께 굴리는 업무로

### 슬라이드 본문

AX는 도구 도입만으로 완성되지 않습니다. 업무 방식도 함께 바뀌어야 합니다.

기존 방식은 사람이 대부분의 단계를 직접 수행하는 방식이었습니다.

```text
자료 수집 → 정리 → 분석 → 초안 작성 → 수정 → 보고서 작성
```

앞으로의 방식은 달라질 수 있습니다.

```text
사람: 목표와 기준 제시
AI: 초안과 반복 작업 수행
사람: 검토, 판단, 책임
```

이 변화에서 사람의 역할은 사라지는 것이 아니라 바뀝니다. 직접 처리자에서 업무 설계자, 검토자, 책임자로 이동합니다.

특히 Skill 영역에서는 직원 참여가 중요합니다. AX는 DX 전문부서가 MCP를 만들어주는 것만으로 완성되지 않습니다. 현업 직원들이 자기 업무의 기준, 예외, 보고 방식, 검토 포인트를 Skill로 정리하고, 실제로 써보며 개선하는 과정이 함께 가야 합니다.

### 화면 표현

- Before: 사람이 혼자 파일 더미를 들고 뛰는 그림
- After: 사람은 감독·검토, AI는 초안·반복 처리
- 옆에 작은 Skill 제작 cycle: `만들기 → 써보기 → 고치기`
- 문구: `직접 처리자 → 업무 설계자·검토자`

## Slide 12. 결론

### 제목

AX는 모델 구매가 아니라, 업무 실행 체계 구축

### 슬라이드 본문

우리 회사가 집중해야 할 것은 최신 모델 하나가 아닙니다. 모델은 출발점입니다. 실제 AX를 위해서는 모델을 업무 안에서 작동하게 만드는 실행 체계가 필요합니다.

미래에 더 뛰어난 LLM이 나오더라도 Skill과 MCP의 중요성은 사라지지 않습니다. 모델이 더 똑똑해질수록 오히려 우리 회사의 업무 지식과 사내 시스템을 얼마나 잘 연결해두었는지가 성과를 가릅니다.

핵심 과제는 네 가지입니다.

1. 업무 지식을 Skill로 축적한다.
2. 사내 데이터와 시스템을 MCP로 연결한다.
3. LLM·Skill·MCP를 감싸는 Harness를 고도화한다.
4. AI가 도구를 쓰고 결과를 다시 확인하는 closed-loop Agentic Workflow를 만든다.
5. 직원들이 Skill을 만들고, 점검하고, 테스트하고, 수정하는 참여 cycle을 만든다.
6. 사람이 목표와 기준을 제시하고 AI가 수행하며 사람이 검토하는 업무방식을 정착시킨다.

마지막 메시지는 다음과 같습니다.

```text
AI를 도입하는 것에서,
AI와 함께 일하는 방식으로.
```

Tool-use와 closed loop가 없으면 AI는 긴 답변을 잘하는 챗봇에 가깝습니다. Tool-use와 closed loop가 Harness 안에서 안정적으로 돌아갈 때 비로소 AI는 실제 업무를 맡길 수 있는 Agent에 가까워집니다.

그리고 이 Agent가 회사 업무에 맞게 일하려면 Skill이 계속 쌓여야 합니다. MyHarness는 단순히 AI Agent를 쓰는 화면이 아니라, 직원들이 회사의 도메인 지식을 Skill로 만들고 검증하는 플랫폼이기도 합니다.

### 화면 표현

- 큰 구조
- `Harness 안에서 LLM·Skill·MCP가 작동`
- 옆에 사람: `목표·기준·검토·책임`
- 목적지: `AX 실행 체계`
