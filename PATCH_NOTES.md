# OpenHarness 패치 노트

이 문서는 로컬 Git 커밋 기록을 바탕으로, 비개발자도 이해하기 쉽게 날짜별 주요 개선사항만 정리한 것입니다.
테스트 보강, 문서의 작은 문구 수정, 내부 정리 같은 자잘한 변경은 관련 기능 아래에 묶었습니다. 단, 직접 추가한 스킬은 사용자가 할 수 있는 일이 늘어나는 큰 변화라 별도로 드러내고, 자동 생성되는 `learned-*` 스킬은 주요 기능 목록에서 제외했습니다.

표기 기준:

- `원본 OpenHarness`: upstream OpenHarness 또는 다른 contributor의 커밋입니다.
- `Myeongcheol`: Git author가 `Myeongcheol` 또는 `Myeongcheol_Oh`인 커밋입니다.

## 2026-04-25 (Myeongcheol)

이 날의 가장 큰 변화는 OpenHarness가 처음으로 웹 앱의 형태를 갖추기 시작했다는 점입니다. 기존에는 명령줄과 터미널 UI에서 쓰는 도구에 가까웠기 때문에, 대화 기록을 다시 찾거나 생성된 파일을 확인하거나 설정을 바꾸는 일이 화면 안에서 매끄럽게 이어지지는 않았습니다. 그래서 브라우저에서 OpenHarness를 사용할 수 있는 첫 웹 UI를 만들었습니다.

- Bash 사용, 웹 가져오기 실패, 검색 결과 없음처럼 반복적으로 겪은 상황은 자동 학습 기록으로 남겼습니다. 다만 이것은 사용 중 생기는 보조 기록이라, 직접 추가한 스킬과는 구분했습니다.
- README와 OpenHarness 설명 문서를 현재 웹 UI 흐름에 맞게 업데이트했습니다. 문서가 실제 사용 방법과 어긋나지 않도록 맞춘 작업입니다.
- 히스토리 버튼, 첨부 영역, 이미지 미리보기, 모델 설정 화면은 실제 캡처를 보며 확인했습니다. 말로만 정리한 것이 아니라, 깨지기 쉬운 화면을 직접 보면서 조정했습니다.

## 2026-04-29 (Myeongcheol)

이날은 웹 UI를 계속 다듬으면서, 동시에 작업 과정에서 생기는 임시 산출물을 정리한 날입니다. 화면을 좋아지게 만드는 일과, 그 화면을 확인하면서 생기는 로그와 캡처가 저장소를 어지럽히지 않게 하는 일을 같이 처리했습니다.

- 체크리스트 UI를 다듬었습니다. 항목을 확인하고 접고 펼치는 요소는 자주 보게 되는 부분이라, 버튼과 텍스트가 어색하게 튀지 않도록 정리했습니다.
- 체크리스트 컨트롤의 밀도를 낮췄습니다. 화면이 무겁고 투박해 보이지 않도록 패딩과 간격, 버튼 형태를 조금 더 슬림하게 맞췄습니다.
- 체크리스트는 작은 컴포넌트지만 실제로는 디자인 시간이 많이 들어간 부분입니다. 접힌 상태, 펼친 상태, 항목이 많을 때, 설명이 길 때를 각각 보면서 버튼 크기와 줄 간격, 텍스트 무게를 계속 맞췄습니다.
- “예쁘지만 큰 UI”가 아니라 “작업 중 계속 켜 둬도 방해되지 않는 UI”를 목표로 했습니다. 그래서 둥근 모서리나 여백을 과하게 쓰지 않고, 필요한 정보가 차분하게 정렬되도록 조정했습니다.
- 웹 UI와 터미널 UI가 같은 제품처럼 느껴지도록 색, 라벨, 간격, 컨트롤 표현을 맞췄습니다. 기능은 달라도 전체 인상이 따로 놀지 않게 하려는 작업이었습니다.
- 긴 텍스트나 좁은 화면에서 메시지와 패널이 겹치거나 잘리는 문제를 줄였습니다. 실제 사용 중 가장 거슬리는 종류의 UI 깨짐을 줄이는 데 초점을 뒀습니다.
- 4월 26일에 넣었던 큰 디자인 스킬 묶음은 기능은 넓었지만 데모, 음원, 샘플 이미지까지 함께 들어와 저장소가 무거워질 수 있었습니다. 그래서 실제 OpenHarness 작업 흐름에 맞게 더 가벼운 스킬들로 나누는 방향으로 정리했습니다.
- `ui-design-essence` 스킬을 추가했습니다. 새 화면이나 컴포넌트를 만들 때 단순히 보기 좋게 꾸미는 것이 아니라, 목적, 사용자, 밀도, 시각적 위계를 먼저 정하고 일관된 디자인 기준으로 다듬기 위한 스킬입니다.
- `visual-artifact` 스킬을 추가했습니다. 보고서, 대시보드, 인포그래픽, 비교표, 발표용 HTML 같은 결과물을 브라우저에서 바로 열 수 있는 단일 시각 자료로 만들기 위한 스킬입니다.
- `visual-review` 스킬을 추가했습니다. 만든 HTML 결과물이 실제 브라우저에서 깨지지 않는지, 글자가 잘리거나 넘치지 않는지, PDF나 스크린샷으로 내보내도 괜찮은지 점검하기 위한 스킬입니다.
- `playwright-capture` 스킬을 추가했습니다. HTML 페이지를 실제 Chromium으로 렌더링해 스크린샷이나 PDF로 저장하고, 발표 자료나 보고서에 쓸 수 있는 상태로 캡처하기 위한 스킬입니다.
- 이 스킬들은 웹 UI 디자인을 계속 다듬으면서 필요해진 작업 흐름을 반영한 것입니다. 화면을 만들고, 시각 자료로 정리하고, 실제 브라우저에서 검수하고, 캡처까지 이어지는 과정을 OpenHarness 안에서 처리할 수 있게 하는 것이 목적이었습니다.
- 체크리스트 상태, 입력 경계, 소프트 리저브 표시처럼 깨지기 쉬운 부분은 화면 캡처로 확인하면서 조정했습니다.
- 디자인 스킬 묶음을 가져오는 과정에서 큰 데모 파일과 음원, 샘플 이미지가 과하게 들어올 수 있다는 점을 확인했고, 어떤 파일을 실제 저장소에 남기고 어떤 것은 제외할지 정리 기준을 다시 잡았습니다.
- UI 점검을 반복하면서 생긴 로그와 스크린샷이 프로젝트 루트에 쌓이는 문제가 있었기 때문에, 이런 검사 산출물이 Git 변경 목록에 섞이지 않도록 ignore 규칙을 추가했습니다.
- 브라우저 점검용 임시 프로필 폴더도 Git 추적 대상에서 제외했습니다. 캐시와 내부 로그가 대량으로 생기는 폴더라, 기능 변경과 함께 관리할 필요가 없다고 판단했습니다.
- 이미 Git이 추적하던 UI 점검 로그와 임시 브라우저 파일은 인덱스에서 제거했습니다. 앞으로는 실제 파일이 남아 있어도 커밋 대상에 올라오지 않게 하기 위한 정리입니다.
- 같은 문제가 반복되지 않도록 작업 지침도 보강했습니다. 앞으로 화면 점검 로그와 스크린샷은 루트가 아니라 무시되는 로컬 작업 폴더에 모으도록 기준을 세웠습니다.
