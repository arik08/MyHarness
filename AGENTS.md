# MyHarness 작업 지침

- 사용자에게는 존댓말을 사용하세요.
- MyHarness는 사무계 업무를 위한 AI AGENT 시스템입니다.
- 보고서·대시보드·인포그래픽 같은 시각 HTML 산출물의 디자인, 차트, 아이콘, 레이아웃 품질 지침은 `visual-artifact` 스킬에 둡니다. 이 파일에는 같은 내용을 중복하지 마세요.
- UI를 만들거나 수정할 때는 가능한 한 슬림하고 깔끔한 밀도를 선호하세요. 버튼, 카드, 모달, 선택지의 모서리 round는 과하지 않게 절제하고, 불필요하게 두툼한 패딩이나 큰 행 높이는 피하세요.
- UI 툴팁은 브라우저 기본 `title` 툴팁을 쓰지 말고, 앱의 기존 커스텀 툴팁 스타일과 동작을 재사용하세요. 기본 툴팁은 표시 지연과 디자인 불일치가 있어 MyHarness 화면에서는 피하세요.
- 스킬 설명 번역은 원본 `SKILL.md`가 아니라 UI i18n/매핑에서 처리하세요.
- UI 화면 점검용 stdout/stderr 로그, 스크린샷, 임시 브라우저 프로필은 저장이 필요할 때 프로젝트 루트에 흩뿌리지 말고 `.myharness/ui-checks/` 같은 ignore된 로컬 작업 폴더에 모으세요.
- `PATCH_NOTES.md`는 하루에 한 번 수준의 변경 기록 정리용으로 보고, 사용자가 요청했거나 릴리즈/일일 정리처럼 명확히 필요한 경우에만 수정하세요.
- 한글/비ASCII가 들어간 파일을 PowerShell 기본 `Get-Content`/`Set-Content`/`Out-File`로 재저장하지 마세요. 읽을 때는 `-Encoding UTF8`을 명시하고, 수정은 `apply_patch`나 UTF-8을 명시한 도구를 사용한 뒤 `py -3 scripts/utf8_guard.py --changed`로 인코딩 깨짐을 확인하세요.

## Karpathy Guidelines

- 코딩, 리뷰, 리팩터링 작업에서는 Andrej Karpathy식 간결한 엔지니어링 지침을 따르세요. 이 지침은 속도보다 신중함과 정확성을 우선합니다.
- 사용자에게 진행 상황을 설명할 때 특수한 스킬을 사용하는 경우에는 알려도 좋지만, 대부분의 코딩 작업에 기본 적용되는 `karpathy-guidelines`는 굳이 언급하지 말고 실제로 할 작업만 자연스럽게 말하세요.
- 구현 전에 가정을 명시하세요. 해석이 여러 가지이거나 요구가 불명확하면 조용히 추측하지 말고 혼란스러운 지점을 드러내세요.
- 요청받은 범위 안에서 문제를 해결하는 최소한의 코드를 작성하세요. 단일 사용처를 위한 추상화, 요청되지 않은 확장성, 불필요한 설정 옵션을 추가하지 마세요.
- 기존 코드를 수정할 때는 필요한 부분만 외과적으로 변경하세요. 주변 코드 정리, 무관한 리팩터링, 기존 스타일 변경은 사용자가 요청하지 않는 한 하지 마세요.
- 변경한 코드 때문에 새로 생긴 미사용 import, 변수, 함수는 정리하되, 기존에 있던 무관한 죽은 코드는 삭제하지 말고 필요하면 언급만 하세요.
- 작업 목표를 검증 가능한 기준으로 바꾸세요. 버그 수정은 재현 테스트 후 통과시키고, 검증 가능한 명령이나 확인 절차로 완료 여부를 확인하세요.

## Code Graph MCP

- `better-code-review-graph` MCP의 기존 인덱스 조회는 부담이 작으므로 코드 수정, 리뷰, 리팩터링, 버그 조사, 영향 범위 검토에서는 무난하게 자주 사용하세요.
- 여러 파일이 얽힌 작업, 기존 함수·컴포넌트·API의 사용처가 궁금한 작업, 프론트엔드-백엔드 계약이나 상태 흐름을 건드리는 작업에서는 `rg`만으로 결론내기 전에 그래프 query를 먼저 한 번 확인하세요.
- 단일 파일 작업이라도 호출자, import 관계, 테스트 위치, 변경 여파가 조금이라도 애매하면 기존 인덱스에 대해 `callers_of`, `callees_of`, `imports_of`, `importers_of`, `tests_for`, `impact` 같은 저비용 query를 먼저 시도하세요.
- 그래프 조회를 생략해도 되는 경우는 순수 문구 수정, 명백히 독립적인 스타일 한 줄 수정, 이미 열린 파일 안에서 완전히 국소적인 변경처럼 영향 범위가 자명한 작업으로 제한하세요.
- 그래프를 사용할 때는 build/update부터 시작하지 말고, 기존 인덱스에 대한 읽기 query를 우선하세요. 단순 키워드 검색 결과만으로 아키텍처 결론을 내리지 말고, 그래프 query 결과가 유효한 경우에는 실제 파일 내용과 함께 확인하세요.
- 그래프 결과가 없거나 오래됐다고 의심되어도, 작업 안전성이 그래프 최신성에 달려 있지 않다면 build/update로 시간을 쓰지 말고 파일 기반 확인으로 진행한 뒤 한계를 짧게 공유하세요.
- graph build/update는 사용자가 명시적으로 요청했거나, 넓은 리팩터링처럼 최신 그래프 없이는 위험한 경우에만 실행하세요. 한 번 긴 timeout으로 실패하면 자동 재시도하지 말고, ignore 범위나 부분 생성 상태를 점검할 필요가 있다고 보고한 뒤 테스트/타입체크 같은 검증을 우선하세요.
- graph build/update는 MCP tool call 경로가 Codex 앱의 120초 대기 제한에 걸릴 수 있으므로, 긴 timeout이 필요한 경우 MCP `graph` 호출 대신 `uvx --python 3.13 --from better-code-review-graph python -X utf8 -c "from better_code_review_graph.tools import build_or_update_graph; print(build_or_update_graph(full_rebuild=False, repo_root=r'C:\Users\user\Desktop\Documents\Python\MyHarness'))"`처럼 패키지 내부 함수를 직접 실행하세요. 600초 안에도 완료되지 않으면 해당 세션에서는 그래프를 쓰지 말고 파일 기반 확인으로 진행하세요.
- 그래프 MCP는 구현을 막는 장시간 갱신 작업이 아니라면 적극적인 사전 조회 도구로 취급하세요. 읽기 query는 부담이 작으므로 영향 범위 파악에 자주 활용하세요.
